# Mom4AI v0.2 Execution Plan

## Zielbild

v0.2 schließt den kritischen Gap von **evolviertem DAG-Skelett** zu **trainierbarem Child-Modell** und koppelt Resonanz-Rückfluss robust in die Selektion zurück.

Prioritätsreihenfolge:
1. Assembly (DAG -> PyTorch)
2. Training (Child-GPT Pipeline)
3. Resonanz-Klassenlernen (Human vs Agent)

---

## Phase A — Assembly Compiler

### Deliverables
- Deterministischer DAG->`nn.Module` Compiler (MVP)
- Validierung für Topologie, In/Out-Dimensionen, Merge-Strategien
- Referenz-Fixtures für 3-5 Skeleton-Typen

### Exit-Kriterien
- Für alle Fixtures erzeugt der Compiler lauffähige Modelle
- Forward-Pass Smoke-Test ist stabil
- Fehlerfälle liefern erklärende Validierungsfehler

### Metriken
- Compiler success rate (fixtures)
- Compile time pro Skeleton
- Validierungsfehler pro Kategorie

---

## Phase B — Child-GPT Training Pipeline

### Deliverables
- Training-Konfiguration (Tokenizer, Positional Encoding, Checkpointing)
- Minimale Datenschnittstelle + reproduzierbarer Trainingslauf
- Artefakt-Policy für Modelle/Reports

### Exit-Kriterien
- Reproduzierbarer Mini-Train-Lauf auf Fixture-DAGs
- Checkpoints + Metrics konsistent erzeugt
- Pipeline ist per CLI und CI-smoke nutzbar

### Metriken
- Train step stability
- Loss trend (nur Qualitätsindikator, kein SOTA-Ziel)
- Checkpoint integrity rate

---

## Phase C — Resonanz-Klassenlernen

### Deliverables
- Trennung der Resonanzklassen: Human vs Agent
- Guardrails gegen Metrik-Gaming (mind. baseline ruleset)
- Rückkopplung in Selection-Weighting

### Exit-Kriterien
- Ereignisse sind klassifiziert (human/agent/unknown)
- Quality-Gate für Datenintegrität aktiv
- Selektionslogik nutzt Klassensignal nachweisbar

### Metriken
- Class coverage rate
- Konfliktrate (inkonsistente Labels)
- Einfluss auf Selektion (A/B oder Kontrolllauf)

---

## Governance

- Tasks werden atomar geliefert (`tasks/v0_2/*.md`)
- Jede Umsetzung muss akzeptanzkriterien + Tests im Task referenzieren
- Keine Scope-Erweiterung ohne neuen Task
